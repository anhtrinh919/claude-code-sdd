# Design Brief — External Tool Path

Lean brief for handoff to Pencil.dev, Claude Design, Figma, or similar. High-level direction only — the external tool owns component, layout, and interaction decisions.

Do **not** include: specific component names, interaction patterns, layout hierarchy, mobile breakpoints, empty/error state copy. Those belong to the design tool.

---

## Product context

One paragraph from `mission.md`: who uses this, what problem it solves, what success feels like.

## Visual style

Pick a specific direction (e.g. "editorial minimal," "dense operator dashboard," "playful consumer"). State **why** — tie to who uses it and what it should feel like. One paragraph.

## Tone and mood

Three to five adjectives that describe the feel. Each adjective backed by a one-line reason.

## Palette direction

Rough color intent — light/dark bias, saturation level, accent role. Not hex codes. The design tool picks specifics.

## Typography direction

Serif vs. sans, display vs. utilitarian, weight personality. Not font names. The design tool picks specifics.

## References (optional)

If relevant, name 1–3 products whose design feel is close to the target. State which aspect each reference captures (layout density, color, voice, etc.).

## Screen checklist

Numbered list of every screen + state from `requirements.md`'s UI Requirements table. This is the coverage contract — every item must appear in the returned design.

```
1. Home — empty state
2. Home — populated
3. Board — default view
4. Board — editing
...
```
